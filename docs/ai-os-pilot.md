# AI OS Pilot

## Outcome

Turn one existing Linux workstation into a bounded, local-first AI workflow that can:

- retrieve useful project context without broad rediscovery;
- route work between the client's existing AI tools and local models;
- keep Obsidian, repositories and operating runbooks connected;
- expose health and readiness through one monitoring surface;
- produce a concrete acceptance report and restore dry-run.

This is an implementation service. It is not a new foundation model, a replacement operating-system kernel or a promise of unrestricted autonomy.

## Who it is for

- Linux developers who already use several of Codex, Claude, OpenClaw, Ollama, Open WebUI or Obsidian;
- people losing time to repeated context discovery and disconnected AI chats;
- one-person or small engineering workflows that need privacy, auditability and recovery more than another generic chatbot.

## Acceptance criteria

The pilot is accepted only when:

1. agreed control queries retrieve the expected project context;
2. secrets, credentials and browser/auth state do not enter reports or tracked context;
3. the primary UI and monitoring surface are reachable in the agreed local environment;
4. health checks identify the current host state instead of claiming false health;
5. the restore path completes a documented dry-run;
6. the client receives a concise runbook and acceptance report.

## Boundaries

- no passwords, API tokens, cookies, SSH keys or private files in the initial request;
- no unrestricted remote command runner;
- no claim of full autonomy;
- no software-rendering workaround for GPU-backed desktop applications;
- cloud services remain optional and explicit;
- unsupported hardware or non-Linux hosts start with the diagnostic only.
